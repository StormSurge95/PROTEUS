# Plugin Development Guide

## Scope
This guide describes how to add or maintain a Proteus console core plugin.

Current plugin projects:
- `PROTEUS/ProteusNES`
- `PROTEUS/ProteusGBA`

## Prerequisites

1. Access to shared interfaces under `PROTEUS/Proteus/src/shared`
2. CMake build configured through root presets in `PROTEUS/CMakePresets.json`
3. Matching contract versions for `IConsole` and `IDebugger`

## Minimal Plugin Checklist

1. Create/maintain plugin CMake target as a shared library.
2. Export required manifest/core symbols.
3. Include shared contract headers.
4. Emit plugin binary to app-expected plugin output directory.
5. Verify load/create/destroy through tests.

## Export Implementation Pattern

Define exports in a dedicated file (example pattern: `NesCoreExports.cpp`):

1. `PLUGIN_MANIFEST(GetPluginManifest)`
2. `CORE_CREATE(CreateCore)`
3. `CORE_DESTROY(DestroyCore)`
4. optional debugger exports

Keep exported signatures exactly aligned to `PluginExports.h`.

## CMake Target Requirements

Each plugin target should:
- be `SHARED`
- compile required source units for runtime viability
- include shared headers from `PROTEUS/Proteus/src/shared`
- set consistent output paths so runtime discovery can find the library

On Windows, plugin naming should omit default `lib` prefix (`PREFIX ""`).

## Registry Integration

To make a new plugin discoverable:

1. Add a plugin id -> library basename mapping in `PluginRegistry` (`libBaseNames`).
2. Ensure plugin manifest `consoleName` and metadata are accurate.
3. Confirm plugin filename generated by loader matches target output name.

## Testing Requirements

Add or update tests under `PROTEUS/Proteus/src/tests`:
- discovery
- load/unload
- core lifecycle
- debugger lifecycle (if supported)
- negative diagnostics

For negative API tests, use fixture plugin targets under `src/tests/fixtures`.

## Cross-Platform Notes

1. Avoid compiler-specific calling convention assumptions in non-Windows builds.
2. Keep exported symbol names identical across platforms.
3. Validate plugin behavior through CI on Windows/Linux/macOS.

## Common Failure Modes

1. Missing required export symbol
2. Manifest contract mismatch
3. Plugin output path mismatch with registry search paths
4. Prefix/extension mismatch for platform
5. Attempted unload while live core/debugger still exists
